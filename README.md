<h1>1만 시간의 법칙 웹 프로젝트

10,000 Hour Rule – Web Application</h1>

<h2>1. 프로젝트 개요</h2>
본 프로젝트는 ‘1만 시간의 법칙’을 주제로 제작한 정적(Static) 웹 랜딩 페이지입니다.
시각적 요소(시계 배경, 타이틀 이미지, 따옴표 장식, CTA 버튼, 결과 강조 텍스트)를 중심으로 사용자가 목표 분야와 하루 훈련 시간을 입력하는 흐름을 제공하며, HTML/CSS 기반의 레이아웃 설계 및 반응형 퍼블리싱 역량을 보여주는 것을 목적으로 제작되었습니다.

배포: GitHub Pages

목적: 퍼블리싱(레이아웃/타이포/반응형) 구현 역량 증명

<h2>5. 구현 포인트</h2>
<h3>1. 명확한 정보 흐름을 고려한 화면 구성</h3>

사용자가 서비스 목적을 빠르게 이해할 수 있도록<br>
<strong>문제 제시 → 입력 → 결과 확인</strong>의 단계적 흐름으로 화면을 구성하였습니다.<br>
불필요한 요소를 배제하고 핵심 메시지와 입력 영역에 시선이 집중되도록 레이아웃을 설계했습니다.<br>

<h3>2. 시맨틱 마크업 기반 HTML 구조</h3>

header, main, section, footer 등 시맨틱 태그를 활용하여<br>
콘텐츠의 의미와 구조가 명확하게 드러나도록 구현하였습니다.<br>
이를 통해 가독성과 유지보수성을 동시에 고려한 코드 구조를 유지했습니다.<br>

<h3>3. CSS 중심의 레이아웃 설계</h3>

Flexbox를 활용하여 콘텐츠 정렬 및 간격을 일관되게 관리하였으며,<br>
공통 스타일과 컴포넌트 단위 스타일을 분리하여 코드 중복을 최소화했습니다.<br>
전체 페이지에서 통일된 색상, 버튼 스타일, 타이포그래피를 유지하도록 설계했습니다.<br>

<h3>4. 사용자 입력을 고려한 UI 구성</h3>

입력 영역은 사용자가 직관적으로 이해할 수 있도록<br>
placeholder 문구와 문장형 설명을 함께 배치하였습니다.<br>
버튼과 입력 요소는 시각적으로 명확한 구분이 가능하도록 디자인했습니다.<br>

<h3>5. 정적 웹 서비스에 적합한 구조 설계</h3>

서버 의존 없이 동작하는 정적 웹 프로젝트 특성을 고려하여<br>
모든 리소스를 클라이언트 단에서 처리하는 구조로 설계하였습니다.<br>
GitHub Pages 배포 환경에 적합하도록 경로 관리 및 파일 구조를 구성했습니다.<br>

<h3>6. 유지보수와 확장을 고려한 파일 분리</h3>

HTML, CSS 파일을 역할별로 분리하여<br>
향후 기능 추가나 디자인 수정 시 영향을 최소화할 수 있도록 구현했습니다.<br>
작은 규모의 프로젝트이지만 실무 환경을 고려한 구조를 유지하는 데 중점을 두었습니다.<br>
<hr />
<h2>6. 결론 및 느낀점</h2>

본 프로젝트를 통해 정적 웹 서비스 개발의 전반적인 흐름을 경험할 수 있었습니다.
서비스 목적을 명확히 정의하고, 사용자가 자연스럽게 이해할 수 있는 화면 구성과 콘텐츠 배치를 고민하는 과정에서 <strong>기획과 구현의 연결성</strong>이 중요하다는 점을 체감했습니다.<br>

특히 HTML 구조 설계와 CSS 레이아웃 구현 과정에서, 초기 설계 단계에서의 구조적 판단이 이후 수정과 유지보수에 큰 영향을 미친다는 것을 확인했습니다. 단순히 화면을 구현하는 것을 넘어, <strong>코드의 가독성과 확장 가능성</strong>을 고려한 작성이 필요함을 인식하게 되었습니다.

또한 GitHub Pages를 활용한 배포 경험을 통해, 로컬 개발 환경과 실제 배포 환경 간의 차이를 이해하고 정적 웹 서비스의 배포 구조에 대한 실무적인 감각을 익힐 수 있었습니다. 제한된 기간 내에 기획부터 배포까지를 완료하며, <strong>일정 관리와 작업 우선순위 설정의 중요성</strong>도 함께 학습했습니다.

이번 프로젝트는 소규모 개인 프로젝트였지만, 웹 서비스 개발의 기본적인 설계 사고와 구현 역량을 점검할 수 있는 계기가 되었으며, 향후 더 복잡한 기능과 구조를 가진 프로젝트로 확장하기 위한 기반을 다지는 경험이 되었습니다.
